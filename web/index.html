Contract AI that refuses to guess

How It Works

Four stages turn a raw PDF into a verifiable, abstention-capable Q&A system.

1

Typed Graph Extraction

The PDF is chunked and each chunk is processed by an LLM to extract structured nodes like obligations, definitions, conditions, and key facts. Co-occurring nodes within a chunk are connected by edges, creating a structured knowledge graph.

2

Multi-Variant Retrieval

At query time, the system generates multiple versions of the question and retrieves evidence for each. The system keeps overlapping results and high-confidence supporting nodes, reducing noise from any single retrieval path.

3

Answerability Gate

The system combines multiple signals, including LLM judgment and graph structure, to determine whether the evidence is strong enough to answer.

✓ ✗

Grounded Answer or Principled Abstention

If sufficient evidence is found, the system generates an answer grounded in the verified evidence and cites supporting text from the document. If not, it abstains.

Benchmark Results

Evaluated on 500+ questions across 12+ real contracts not seen during development.

Why this matters. In legal settings, a confident wrong answer can cost real money and real trust. This system avoids that by only answering when the evidence is actually there, and refusing to answer when it is not. A well-calibrated "I don't know" is more valuable than a fluent hallucination.

Technical Notes

Graph construction

Contracts are chunked with overlap, then each chunk is passed to an LLM that extracts typed nodes. Node types: definition, obligation, right, numeric, condition, blank, reference. Edges connect co-occurring nodes within the same chunk. The resulting graph is stored and reused across all queries on the same document.

Retrieval stability via query variants

A single query can be ambiguous or phrased in ways that miss relevant clauses. Keeping overlapping results and high-confidence supporting nodes reduces noise from any single retrieval path.

Topology scoring

Beyond LLM judgment, a topology signal quantifies how well-connected the retrieved nodes are in the graph. High connectivity (short paths, dense neighborhoods near retrieved nodes) correlates with the document genuinely covering the question topic. The threshold adapts based on the structure and coverage of the retrieved nodes.

Answerability decision logic

The system combines multiple signals, including LLM judgment and graph structure, to determine whether the evidence is strong enough to answer.

About

I'm a student builder, and this project started with a simple question: what would legal AI look like if it actually cared about being right, not just sounding right?

I started by experimenting with standard RAG, trying to see if better retrieval could improve accuracy. At first it was general, but I focused on legal documents because the problem became much clearer there. In law, answers need to be backed by exact text, so it was easy to see when the system was guessing versus actually supported.

As I kept testing, I realized the real problem was not just retrieval, but knowing whether the answer is actually in the document, not just something that sounds topically similar.

So instead of building a system that always responds, I built one that checks whether the evidence is really there, and if it is not, it just does not answer.

I used ChatGPT as a build partner the whole way through, designing the system, debugging retrieval, and testing it across hundreds of questions. What you see here is a real system tested on real contracts.

Right now it works on one document at a time, but the goal is to extend this across multiple documents while keeping the same idea: only answer when the evidence is actually there.
